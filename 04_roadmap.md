# Продуктовий роадмеп · Moment Planner (Фаза 1)

Це контракт між продуктовим рішенням і розробкою. Він передається в Claude Code як основа Фази 2 (побудова MVP). Усе, що тут написано, — рішення вже прийняті. Claude Code не переобдумує стратегію, а реалізує її. Якщо в процесі щось із цього документа неможливо або суперечливо — зупинитись і запитати, не вигадувати.

Мова продукту та вводу — **українська**. Інструмент для українського ринку.

## 1. Що будуємо і чого не будуємо

### Стратегія: вузька

Один завершений наскрізний сценарій цінніший за багато незавершених функцій. Ядро доводимо до бездоганності. Одна надбудова поверх нього. Решта типів умов — видимий каркас без коду.

### Жорстке правило черговості

**Гео не починається, поки ядро на часі не проходить наскрізь і стабільно.** Спершу повністю робочий Capture → Inbox → Today. Тільки після цього — гео. Не паралельно.

Якщо часу до дедлайну вистачає лише на ядро — здаємо ядро + усі надбудовні типи (включно з гео) у вигляді замкнених елементів UI. Це все одно завершена, чесна здача. Недороблене ядро перекреслює будь-яку елегантність зверху.

### Scope: три категорії

**Реалізуємо по-справжньому:**
- Ядро на часі: Capture → Inbox → Today, наскрізь.
- Полиморфний рушій умови (див. розділ 2), реально працює для типу `time`.
- Парсинг сирого тексту в структуровані наміри через Claude Haiku.
- Гео як другий тип умови — **тільки після стабільного ядра**.

**Каркас без коду (видно в UI, логіки нема):**
- Наступні типи умов: курс валют, якість повітря, розклад, астрономічні цикли. З'являються як недоступні елементи з позначкою «скоро». Доводять масштабованість, коштують нуль дня роботи.

**Не чіпаємо взагалі:**
- Датчики тіла (пульс, сон) — недоступні у вебі.
- Пуш-сповіщення на закритому застосунку — обмеження браузера, чесно визнаємо.
- Спільний доступ, командна робота, обмін.
- Складна система тегів і проєктів. Свідомо ні — будуємо протилежність складу.
- **Повторювані/рекурентні наміри** («щопонеділка»). Модель одноразова. Рекурентність — окрема велика механіка, поза MVP.
- **Ручне редагування полів наміру.** Дії лише: виконано / відпустити / перемкнути «в сьогодні». Цього достатньо, щоб коригувати план (закриває критерій 4). Повне редагування — YAGNI.

### Технічні межі (задані воркшопом)

- Дані демо зберігаються **локально в браузері** (localStorage). Окремий сервер бази не потрібен.
- API-ключ Anthropic живе **на сервері** (env-змінна на Vercel `ANTHROPIC_API_KEY`), ніколи в браузері чи git. Парсинг іде через серверний API-route Next.js — він проксі між браузером і Anthropic.
- Відповідь моделі **не приймати на віру** — валідувати й чистити формат.
- **Mobile-first.** Весь сценарій проходиться однією рукою в транспорті.
- Модель парсингу — **Claude Haiku** (`claude-haiku-4-5`), швидка й дешева.

## 2. Схема даних — серце продукту

Це головний контракт. Усе інше — надбудова над цією моделлю.

### Сутність: намір (Intent)

Не «задача». Намір, що чекає на свою обставину.

```
Intent {
  id:        string            // унікальний
  text:      string            // сам намір, як його розпарсив AI
  priority:  "high" | "medium" | "low"
  status:    "open" | "done" | "released"   // ТІЛЬКИ людські рішення (див. нижче)
  condition: Condition         // умова доречності — див. нижче
  createdAt: timestamp
}
```

**Критично — статус vs активність (не плутати дві моделі):**
- `status` зберігає **лише дії людини**: намір активний (`open`), людина виконала (`done`), людина відпустила (`released`). Це записані факти.
- Чи намір **зараз доречний** (чи «виринув») — **не статус, а похідне**: обчислюється на льоту перевіркою умови в поточний момент. Не зберігається. Немає записаного `surfaced`, який може «протухнути» — є лише результат перевірки `умова.активна(зараз) → true/false`.
- Це знімає конфлікт «намір surfaced, а вікно вже минуло»: доречність рахується щоразу заново, а не консервується.

### Ключова абстракція: умова (Condition)

Умова — **полиморфний об'єкт**, а не захардкоджене поле «час» чи «дедлайн». Це вся різниця між «час — перший тип умови» і «час прибитий цвяхами, а погоду потім приклеїмо збоку».

```
Condition {
  type:  "time" | "location" | "weather" | ...   // тип умови
  value: object                                   // значення, залежить від типу
}
```

Для ядра `type` завжди `"time"`. Значення для часу:

```
value (для type="time") {
  kind:     "datetime" | "date" | "weekday" | "daypart"
  at:       timestamp | null      // конкретний момент/дедлайн
  weekday:  string | null         // день тижня — ОДНОРАЗОВО: "найближчий вказаний"
  daypart:  "morning" | "afternoon" | "evening" | null
}
```

**`weekday` — одноразовий, не повторюваний.** «У понеділок» = найближчий понеділок, один намір. **Рекурентності («щопонеділка») в MVP немає** — модель одноразова (`done`/`released` спрацьовують раз), повтори вимагали б окремої механіки генерації та історії виконань. Це свідомо поза ядром (див. «Не чіпаємо»).

**`daypart` впливає лише на порядок, не на появу.** Намір з `daypart: "morning"` — у Today **весь день**, але сортується вище зранку. Today = денний список, не «момент зараз» (див. розділ 3, гранулярність Today).

Значення для гео (реалізуємо у другу чергу):

```
value (для type="location") {
  placeName: string             // named-місце: "Львів", "дім", "робота"
  lat:       number
  lng:       number
  radiusM:   number             // радіус спрацювання в метрах
}
```

**Правило:** додати новий тип умови = додати новий модуль-перевіряч, а не переписати ядро. Це має бути видно в архітектурі.

### Статуси і їхній рух

Збережені статуси — тільки людські рішення:

```
open      → намір живий. Лежить тихо, чекає на умову. Не тисне, не рахується як борг.
done      → людина виконала.
released  → людина відпустила (момент минув / неактуально). Повернути в open можна.
```

Поверх статусу — **похідна активність** (не зберігається): для `open`-наміру щоразу рахуємо, чи його умова доречна зараз. Якщо так — намір «виринув» і потрапляє в Today. Якщо ні — тихо лежить.

### Механізм відпускання (тригер — відкриття, не годинник)

**Проблема:** у вебі немає фонового процесу. Сайт не працює, коли закритий, тож ніхто не переведе намір у `released` опівночі автоматично. Це та сама межа вебу, що й з гео.

**Рішення:** відпускання — не автоматичне за таймером, а **при наступному відкритті застосунку**. Коли людина заходить, продукт перераховує наміри й ті, чий момент об'єктивно минув (напр. часова умова «сьогодні», а сьогодні вже інший день), показує **приглушено** з м'якою опцією «відпустити / повернути». Система не гризе — вона тихо каже «оце вже минуло, прибрати?». Рішення лишається за людиною, дефолт — відпустити, а не тягнути далі докором.

### Три архітектурні вимоги (інакше доведеться переписувати ядро)

Ці три речі — не про модель даних, а про рантайм. Модель без них — косметика, яку роблять усі й потім дивуються рефактору.

1. **Асинхронна перевірка умови.** Час резолвиться миттєво (глянув на годинник), але гео і погода — це зовнішній виклик. Перевірка умови має бути асинхронною з першого дня, навіть коли час резолвиться миттєво. Інакше додавання гео/погоди ламає синхронний цикл.

2. **Контекст передається ззовні, не читається всередині.** Часу контекст не потрібен, гео потребує поточну позицію. Перевіряч отримує контекст (поточний час / позицію / погоду) параметром, а не лізе по нього сам усередині себе. Часу цей параметр не потрібен — але сигнатура вже готова під гео.

3. **Переоцінка на тригер, не одноразовий підрахунок.** Спокуса порахувати Today один раз при відкритті — головна пастка. Гео і погода спрацьовують у непередбачуваний момент. Піднімання намірів має бути повторюваною оцінкою на тригер, а не одноразовим сортуванням при старті.

## 3. Три екрани ядра

Наскрізний потік: brain dump → структуровані наміри → план на сьогодні.

### Capture — «Що в голові?»

- Велике поле вводу + помітна кнопка мікрофона. Диктуєш або пишеш усе підряд, потоком.
- Голос — через браузерне розпізнавання мовлення (Web Speech API), українською. Поки людина говорить, видно, що продукт її чує.
- Голос — апгрейд поверх тексту. **Спершу бездоганний текстовий потік, голос додається зверху.** Краще робочий текст, ніж зламаний мікрофон.
- Одна дія між «подумав» і «записано» — мінімум тертя.
- Кнопка «Спробувати приклад» — заповнює поле готовим brain dump'ом, щоб журі одразу побачило продукт у дії, не вигадуючи текст на ходу.

**Чесна межа голосу — та сама прозорість, що з гео.** Web Speech API українською надійно працює по суті лише в **Chrome**. На iOS Safari — капризний або відсутній. Тому:
- **Демо голосу пишемо в Chrome** (свідомий вибір браузера, називаємо вголос у Loom).
- **Якщо голос не підтримується — мікрофон просто ховається**, поле тексту працює завжди. Ніякого «зламаного мікрофона» перед журі.
- **Текстовий потік — фундамент, що працює на будь-якому пристрої й браузері**, зокрема на iOS. Навіть якщо голос десь не встане, продукт лишається повним.
- Ризик, який це знімає: не записати Loom з айфона, натиснути мікрофон — і отримати тишу. Фінальний прогін голосу — тільки в перевіреному браузері.

### Inbox — розбір і підтвердження

- AI парсить надиктований потік і розкладає на окремі наміри з пріоритетом і часовою умовою.
- **Момент підтвердження — критичний.** Одразу після розбору видно, як AI зрозумів кожен намір: сам текст, пріоритет, умову. Людина бачить розбір **до збереження** і може виправити в один рух.
- Це не лише зручність — це довіра. Помилка розпізнавання, яку тихо зберегли, — головна причина, чому кидають голосовий ввід.
- Тут же живе **родина типів умов** (див. розділ 6) — видно, що продукт розпізнав тип, і що типів є більше.

### Today — план на сьогодні

- Наміри, чия часова умова справджується сьогодні. Це і є «план на сьогодні», сформований з беклогу.
- Це core-екран за критерієм завдання. Тут — обов'язковий список задач на сьогодні. Читабельний, теплий, але список.
- Задача тапається пальцем (виконати / відпустити / прибрати з сьогодні) — mobile-first, одна рука.

**Гранулярність — денна, не моментна.** Today = «на сьогодні» (весь день), а не «зараз». Намір з `daypart: morning` не зникає опівдні — він у списку весь день, просто вище зранку. «Момент зараз» (намір з'являється/зникає по годинах) — надлишкова складність, що суперечить самій ідеї «плану на день». Виняток — гео/погода (розділ 5): вони таки моментні, і саме тому перевірка умови async та переоцінюється на тригер (розділ 2, вимоги 1 і 3).

**Сортування Today:** за пріоритетом (high → medium → low), усередині пріоритету — за часом дня (`daypart`/`at`: morning → afternoon → evening → без часу). Порядок задано явно, щоб Code не вигадував свій.

### Порожні стани — обов'язково

Перше, що бачить журі — порожній продукт. Абсолютно порожній екран читається як баг.
- Capture порожній → запрошення надиктувати перший намір, приклад-підказка.
- Today порожній → не «нема задач», а тепле пояснення, що тут з'явиться план, коли надиктуєш.

## 4. Контракт парсингу (Claude Haiku)

### Вхід

Сирий текст українською — усе, що людина надиктувала/написала потоком. Може містити кілька намірів, розмовну мову, обірвані фрази.

### Вихід

**Тільки JSON, без преамбули, без markdown-огорожі.** Масив намірів у полиморфній формі:

```json
[
  {
    "text": "купити квитки на потяг",
    "priority": "high",
    "condition": {
      "type": "time",
      "value": { "kind": "date", "at": "2026-07-20T00:00:00", "weekday": null, "daypart": null }
    }
  }
]
```

### Правила парсингу для ядра

- Для ядра `condition.type` **завжди** `"time"`. Гео вмикається розширенням словника пізніше, не зараз.
- Якщо намір не має явного часу — модель ставить розумний дефолт (наприклад, `daypart` або «сьогодні»), а не лишає порожнім. Але не вигадує точну годину, якої людина не казала.
- Пріоритет модель виводить з формулювання («терміново», «не забути обов'язково» → high), за замовчуванням `medium`.

### Валідація (не вірити моделі на слово)

- Розпарсити відповідь як JSON. Якщо не парситься — не показувати сире, а обробити помилку м'яко (повторний запит або чесне повідомлення).
- Перевірити схему: кожен намір має `text`, `priority` з дозволених значень, `condition.type` з дозволених, `value` відповідний типу.
- Відкинути/полагодити криві поля, а не зберігати сміття.

### Мова

Промпт для Haiku і приклади в ньому — **українською**. Модель має розбирати українську розмовну мову як основну.

### Архітектура парсингу (єдиний серверний код)

- Один роут: **`POST /api/parse`**. Приймає сирий текст + **сьогоднішню дату** (щоб «до п'ятниці», «завтра» резолвились відносно правильного дня). Викликає Haiku, повертає провалідований масив намірів.
- API-ключ (`ANTHROPIC_API_KEY`) — тільки в env на Vercel, ніколи в браузері чи git. Роут — проксі між клієнтом і Anthropic.
- Клієнт після відповіді додає наміри в беклог у localStorage. Жодного іншого серверного коду немає.

### Обробка помилок (текст користувача НІКОЛИ не втрачаємо)

- **Невалідний JSON від моделі** → схема-валідація, нормалізація (битий `priority` → `medium`, битий час/дата → порожньо), відкидання сміття. При повному провалі — дружня помилка, **введений текст лишається в полі**.
- **Збій мережі/API** → повідомлення + збережений текст + повторна спроба одним тапом.
- **Порожній / беззмістовний ввід** → м'яко просимо надиктувати більше, не кидаємо помилку.
- **Web Speech не підтримується** → мікрофон ховається, текст працює завжди.

### Тести (прагматично, не заради покриття)

- Юніт: нормалізація відповіді моделі — брудний/частковий JSON → чисті наміри.
- Юніт: розв'язання відносних дат («завтра», «до п'ятниці») від переданої сьогоднішньої дати.
- Рендер: порожній стан і фільтр Today.

### Дані та міграції

- localStorage — **демо-scope**. Схема стабільна на час завдання; міграцій немає. Зміна схеми = чистий localStorage (не тягнемо сумісність зі старими записами).

## 5. Гео — друга черга (тільки після стабільного ядра)

Концептуальний герой продукту: намір чекає не години, а того, що людина опиниться в правильному місці. «Як буду у Львові — зайти в каву на Каві».

### Що реалізуємо по-справжньому

- Геоумова в моделі: named-місце + координати + радіус (див. розділ 2).
- Логіка спрацювання: порівняти поточну позицію з умовою, випливти намір при збігу.
- Named-місця — **демо-набір**: кілька заздалегідь відомих точок з готовими координатами (напр. Львів, дім, робота — ті, що прозвучать у демо). Не геокодування всієї планети — це рядок у дорожній карті, не в MVP.
- Парсинг: Haiku розпізнає named-місце у фразі й повертає `type: "location"` зі значенням place.

### Чесна межа вебу — називаємо вголос

Фонових геонагадувань у браузері **не існує**. Браузер не будить сайт, коли людина фізично приходить у точку. Тому:
- Логіку умови реалізуємо повністю.
- Саме **спрацювання на демо** відтворюємо через перемикач «я зараз тут» (або разову перевірку позиції через браузерну геолокацію).
- Про це говоримо **прямо, не ховаючи**. Це усвідомлене рішення й межа вебу, названа вголос, — не недороблення. Рідний застосунок — у дорожній карті.

Прозорість тут працює на критерій «продуктове мислення»: назвати межу сильніше, ніж замаскувати.

## 6. Родина типів умов в UI — демонстрація масштабу

Мета: журі має **побачити очима**, що продукт — рушій доречності, а не планер на годиннику. Дешево, під критерій 2.

### Де живе

У моменті підтвердження розбору (Inbox). Тип умови показується не як кнопка-інструмент, яку людина натискає, а як **те, що продукт розпізнав** у сказаному. Людина не «бере погодний інструмент» — вона каже «коли піде дощ», а продукт відповідає: почув намір X, умова — погода.

### Як виглядає

- **Живі типи:** час (ядро), гео (друга черга). Впізнавані з півпогляду, але явно однієї родини.
- **Замкнені типи:** курс валют, якість повітря, розклад, цикли. Приглушений ряд із позначкою «скоро». Нуль логіки за ними — disabled-елемент із підписом.

На демо це читається за дві секунди: кілька типів працюють, ще кілька в тій самій формі чекають → це рушій, а не набір фіч.

### Межа

Родина типів — **після стабільного часового ядра**. Замкнені елементи не потребують коду. Форму родини закладаємо, другий рушій (гео) будуємо тільки коли ядро бездоганне.

## Ознака готовності Фази 2

**Мінімум (обов'язково):** Людина надиктовує/пише хаос українською → AI розкладає на наміри з часом і пріоритетом → бачить і підтверджує розбір → Today показує план на сьогодні. Наскрізь, стабільно, на мобільному, за живим URL.

**Повна версія (якщо вистачить часу):** те саме + гео як другий тип умови зі своїм спрацюванням через перемикач + видима родина типів із замкненими наступними.

## Що тримати в голові весь час (макро-принцип)

Продукт-менеджер із розробником, який ніколи не втомлюється. Робота — рішення (що будуємо / що викидаємо / що достатньо добре), не код. Порядок незмінний: спершу ядро наскрізь, тоді — надбудова. За критерієм 1 недороблене ядро перекреслює будь-яку елегантність зверху.
